# vaBench Release v2

`benchmark-vabench-release-v2` is the cleaned release-candidate benchmark tree.
It keeps `benchmark-vabench-release-v1` intact for provenance and migrates tasks
only after their agent-visible prompt boundary has been audited.

## Why v2 Exists

v1 contains useful benchmark assets, but some public prompts mix agent-facing
task statements with internal benchmark metadata, checker names, and repair
diagnostics. v2 fixes that by making the prompt boundary explicit and
machine-checkable.

## Required Task Layout

Each v2 task form uses this structure:

```text
forms/<form>/
  public/
    agent_visible_spec.md
    support/
  agent_prompt.md
  agent_visible_files.json
  task_release_card.json
  private/
    invisible_checker_config.yaml
    invisible_spec_checker_map.json
    gold/
```

For the human-facing layer map and file responsibilities, see
`TASK_FORM_LAYOUT.md`.

Visibility rules:

- `public/agent_visible_spec.md` is the canonical agent-visible functional
  contract. It is the shared public source that both prompt wording and checker
  intent must trace to.
- `agent_prompt.md` is a short task wrapper rendered with
  `public/agent_visible_spec.md`.
- `public/support/*` may be rendered only when allowlisted by
  `agent_visible_files.json`.
- `task_release_card.json`, `private/*`, checker details, root causes, sample
  windows, and gold fixed answers are never agent-visible.
- `task_release_card.json` is the single private task card. It combines release
  metadata with former internal provenance, diagnostics, and checker contract
  notes.
- `private/invisible_spec_checker_map.json` is evaluator-only; it links public
  requirements to private checks and runner functions and stores private
  forbidden-leak terms so humans can audit drift without exposing checker IDs or
  thresholds to agents.
- Current v2 tasks use the agent/invisible names above. Import scripts may keep
  backward-compatible readers for historical drafts, but release-candidate task
  directories should not use legacy draft filenames.
- Generic runner rules may be appended only outside task-specific private data.

## Evaluation Protocol

The current v2 release-candidate score track is `one_shot`: an agent receives
`agent_prompt.md`, `public/agent_visible_spec.md`, and allowlisted public support artifacts,
then submits one final answer. Bugfix and composed-flow tasks therefore must put
the coarse, agent-actionable behavior envelope in `public/agent_visible_spec.md` rather than
relying on hidden checker feedback.

Checker implementations may use private thresholds and sample windows, but their
public behavioral intent must be auditable against
`public/agent_visible_spec.md` through `private/invisible_spec_checker_map.json`.
Exact checker IDs, private threshold values,
root causes, and gold implementations remain hidden.

`private/invisible_checker_config.yaml` is not just documentation: it declares
the executable checker task id consumed by the evaluator and stores task-local
syntax, observables, thresholds, and private policy parameters.

Feedback-style debug evaluation is deferred to a future separate agentic track.
That track may expose sanitized compile/simulation/checker summaries over a
bounded number of repair rounds, but its `pass@K` results must not be mixed with
the one-shot denominator.

## Release-Level Surrounding Settings

The surrounding benchmark settings are unified at the v2 release level. They
should not be hand-edited per task:

- `config/release_config.json` is the single release-local configuration entry.
  It defines task discovery, output paths, prompt visibility rules, score policy,
  evaluation protocol, and evaluator backend roles.
- `MANIFEST.json`, `MANIFEST.csv`, and `MANIFEST.md` index migrated v2 task
  forms.
- `EVALUATOR.json` and `EVALUATOR.md` define the v2 evaluator contract and
  source-of-truth reports.
- `reports/score_denominator_manifest.json` records which migrated rows are
  score-enabled. Fresh v2 EVAS/Spectre certification is required before a row is
  counted.
- `reports/prompt_boundary_audit.json` records the machine-checkable
  public/private prompt-boundary audit.
- `reports/spec_checker_map_audit.json` records the machine-checkable alignment
  audit across `public/agent_visible_spec.md`,
  `private/invisible_checker_config.yaml`, spec-checker maps, and runner checker
  functions.

Per-task migration should only add or update the task-local files under
`forms/<form>/`: `public/agent_visible_spec.md`, `agent_prompt.md`,
`agent_visible_files.json`, `task_release_card.json`, `public/support/*`, and
`private/*`. The release-level indexes and reports are
regenerated by script.

## Replacement Boundary

v2 is intended to be replaceable as a release root. Future runners should accept
either:

- `--release-root benchmark-vabench-release-v2`
- `--config benchmark-vabench-release-v2/config/release_config.json`

The v2 runtime path should not depend on v1 package files. v1 references are
kept only under migration/provenance fields so we can audit where a task came
from without using v1 as an evaluator input.

## Validation

Run from the repository root:

```bash
python3 benchmark-vabench-release-v2/scripts/build_package_indexes.py \
  --root benchmark-vabench-release-v2 \
  --config benchmark-vabench-release-v2/config/release_config.json

python3 benchmark-vabench-release-v2/scripts/audit_prompt_boundaries.py \
  --root benchmark-vabench-release-v2 \
  --config benchmark-vabench-release-v2/config/release_config.json \
  --output benchmark-vabench-release-v2/reports/prompt_boundary_audit.json

python3 benchmark-vabench-release-v2/scripts/audit_spec_checker_map.py \
  --root benchmark-vabench-release-v2 \
  --config benchmark-vabench-release-v2/config/release_config.json \
  --output benchmark-vabench-release-v2/reports/spec_checker_map_audit.json
```

Render the agent-visible prompt for inspection:

```bash
python3 benchmark-vabench-release-v2/scripts/render_agent_prompt.py \
  benchmark-vabench-release-v2/tasks/CT04_baseband_signal_conditioning/vbr1_l1_first_order_lowpass/forms/bugfix/agent_visible_files.json
```

The audit verifies that `agent_prompt.md` + `public/agent_visible_spec.md` and
rendered agent prompts do not contain task-specific forbidden leak phrases declared in
private spec-checker maps. It also checks declared public contract anchors so
one-shot checker targets are grounded in the agent-visible public
specification.
